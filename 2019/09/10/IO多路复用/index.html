IO多路复用

多路复用基于事件驱动来实现，事件驱动是多路复用的核心机制

用来解决单次请求阻塞的问题，例如等待处理read请求，阻塞在哪里

Unix网络编程中，这么定义IO多路复用，内核一旦发现

IO 分类

阻塞式IO 最常见的等待那种

select （阻塞）

int select(int nfds, fd_set *readfds, fd_set *writefds, fd_set *exceptfds, struct timeval *timeout);

nfds 为最大的文件描述符值 +1

readfds 某些文件描述符所指向的 socket 已经有数据可读或者数据 EOF

writefds 某些文件描述符所指向的 socket 是否可写数据了

exceptfds 某些文件描述符所指向的 socket 出现异常

1. fd_set 文件描述符的集合 （数组） 会将其拷贝到内核态，readset在内核态轮训检查
2. 数据结构大小有限  使用的是bitmap 存储fd
3. 当有数据的来，会将fd_set 置位，select返回，然后再根据置位来查找文件描述符
4. 当没有数据，会阻塞。

poll

int poll (struct pollfd *fds, unsigned int nfds, int timeout);

struct  pollfd  {

  int  fd;

  short  events;

  short  revents;

};

pollfd 的结构如下， 每个 fd 都有对应的监听事件 events，和就绪返回的事件 revents，现在 fd 的大小是 int 最大值了

select VS poll:

poll 不需要每次都重新构建需要监控的 fd set，但还是会有引起上下文切换的内存 copy

poll 不需要像 select 那样需要用户计算 fd 的最大值 +1，作为 select 函数的第一个参数

poll 减少了 fd 的遍历，在 select 中监控的某 socket 所对应的 fd 值为 1000，那么需要做 1000 次循环

poll 解除了 select 对于 fd 数量 1024 的限制

poll 在 unix 下不支持

epoll

有数据，进行数据重排，放到前面

while(1){
  puts("round again");
  nfds = epoll_wait(epfd, events, 5, 10000);

//nfds 返回的即为多个前面的有数据的fd

for(i=0;i<nfds;i++) {
        memset(buffer,0,MAXBUF);
        read(events[i].data.fd, buffer, MAXBUF);
        puts(buffer);
}

}
